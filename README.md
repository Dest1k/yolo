# YOLO Detector

Детекция объектов в реальном времени на **Android** и **Desktop** (Linux / Windows / macOS).  
Загрузи любую модель YOLOv5/v8/v10 в формате NCNN, ONNX или PT, направь на камеру или MJPEG-поток — и получи живые боксы с счётчиком объектов по классам на полной частоте кадров.

## Ключевые фишки

### MJPEG-сервер (трансляция с боксами)
Прямо из экрана камеры можно включить HTTP-сервер.  
Любой браузер, VLC или другой экземпляр приложения открывает `http://<IP-устройства>:8080` и видит аннотированное видео в реальном времени.  
Наложение боксов происходит на каждом захваченном кадре — сервер работает на полной скорости камеры (~30 fps) независимо от того, как быстро работает нейросеть.

### Входящий MJPEG-поток
Вместо камеры телефона можно подать любой HTTP MJPEG-поток (IP-камера, другой телефон, веб-камера с компьютера).  
Тот же пайплайн инференса обрабатывает входящие кадры — удобно, когда нужно анализировать стационарную камеру без участия телефона.

### Умная запись (Smart Record)
Два режима видеозаписи:
- **Always** — пишет непрерывно пока кнопка активна.
- **Smart** — стартует автоматически в момент появления любого объекта, останавливается через 3 секунды после того, как объекты пропали. Никакого ручного управления.

Готовые ролики сохраняются в стандартную папку загрузок ОС — `Downloads/YoloDetector`.

### Ручной режим камеры
Кнопка ручного управления (FAB слева внизу) открывает панель со всеми инструментами, которые поддерживает выбранная камера: зум, яркость (EV), ручная экспозиция (ISO + выдержка) и ручной фокус. Панель строится динамически — недоступные на конкретном сенсоре контролы просто не показываются.

### Headless-режим для одноплатников (Desktop)
На Raspberry Pi и других ARM-платах можно запускать без GUI и монитора, прямо по
SSH: приложение крутит детекцию и постоянно вещает MJPEG-поток в локальную сеть.
Настраивается переменными окружения, ставится в автозапуск через systemd. См.
раздел [«Одноплатник (Raspberry Pi)»](#одноплатник-raspberry-pi--headless-режим).

### GPU-ускорение на Desktop
Десктопное приложение перебирает провайдеры выполнения по порядку при запуске:
1. **CUDA** — NVIDIA на Linux / Windows
2. **DirectML** — AMD или NVIDIA на Windows
3. **CPU** — всегда работает, никаких специальных драйверов

Режим можно зафиксировать вручную (AUTO / CUDA / DirectML / CPU) в боковой панели. После загрузки модели показывается активный провайдер.

### Поддержка PyTorch `.pt` моделей (Desktop)
Помимо ONNX, десктопное приложение загружает TorchScript-экспортированные `.pt` файлы через DJL.  
Поддерживаются YOLOv5 anchor-free выход `[1, N, 4+C]` и YOLOv10 NMS-free выход `[1, N, 6]`.

## Поддерживаемые форматы моделей

### NCNN — только Android
- Пара файлов `.param` + `.bin`
- YOLOv5/v6/v7 (anchor-based), YOLOv8/v9/v11 (сырая DFL-голова — декодируется на устройстве), YOLOv10 (NMS-free)
- CPU или GPU (Vulkan) через нативную библиотеку NCNN
- Парсер сам определяет формат выхода по числу классов: сырой DFL-головой (64+nc), декодированными боксами (4+nc) или с objectness (4+1+nc)

> **Библиотека моделей** содержит выверенный набор готовых моделей (YOLO11n, YOLOv8n, YOLOv5s). При выборе модели автоматически подставляются оптимальные пороги, число потоков под CPU устройства и имена классов COCO — настраивать ничего не нужно.

### ONNX — Android + Desktop
- Один файл `.onnx`
- Desktop сам определяет раскладку выхода: **NMS-free** `[1, N, 6]` (YOLOv10), **anchor-free** `[1, 4+nc, N]` (YOLOv8/v9/v11) и **с objectness** `[1, 5+nc, N]` (YOLOv5/v6). Размер входа и normalized/pixel-координаты определяются автоматически. Для верного разбора v5/v6 важно правильное число классов — задаётся `YOLO_CLASSES` или числом строк в `YOLO_LABELS`.
- Кастомные классы: `YOLO_LABELS=labels.txt` (по имени класса на строку) вместо встроенных меток COCO
- Android: доступно ускорение через NNAPI
- Desktop: CUDA EP (NVIDIA) + DirectML EP (AMD/NVIDIA Windows); автоматический откат на CPU

### PyTorch TorchScript — только Desktop
- Один файл `.pt`, экспортированный через `model.export(format="torchscript")`
- YOLOv5 выход `[1, N, 4+C]` и YOLOv10 выход `[1, N, 6]`
- GPU через CUDA 12.1 (NVIDIA); откат на CPU

## Android

### Требования
- Android 8.0+ (API 26)
- Разрешение камеры (для режима камеры)
- Разрешение сети (для работы со стримами)

### Сборка
```bash
git clone https://github.com/dest1k/yolo.git
cd yolo
./gradlew :app:assembleDebug
# APK → app/build/outputs/apk/debug/app-debug.apk
```

### Использование

1. **Загрузи модель**
   - Нажми **Библиотека моделей** — скачать готовую NCNN-модель одним тапом, или
   - **Обзор** рядом с `.param` / `.bin` для NCNN-файлов, или
   - **Обзор** рядом с ONNX для `.onnx`-файла

2. **Настрой** через **Настройки модели**:
   - Версия YOLO (v5 / v8 / v10), размер входа, количество классов
   - Пороги Confidence и NMS, количество потоков CPU, GPU
   - Имена выходных слоёв (или автоопределение через **Определить выходы**)

3. **Входящий стрим** (опционально) — введи HTTP MJPEG URL в поле стрима, чтобы использовать сетевую камеру вместо встроенной

4. Нажми **Запустить камеру** (или **Запустить стрим**)

### Управление в режиме камеры

| Кнопка | Действие |
|--------|----------|
| **MJPEG** | Включить / выключить встроенный MJPEG-сервер. URL отображается под кнопкой. |
| **720p** | Переключить разрешение: 480p → 720p → 1080p |
| 📷 FAB | Сохранить текущий кадр с боксами в галерею |
| 🎥 FAB | Начать / остановить запись (режим Always) |
| ⚙ FAB | Открыть настройки прямо во время работы |
| 🔍 FAB (слева внизу) | Включить / выключить режим Smart Record |
| 🎚 FAB (слева внизу) | Открыть панель ручного управления камерой (зум, EV, ISO, выдержка, фокус) |
| ↔ FAB | Переключить фронтальную / основную камеру |

## Desktop

### Требования
- JDK 17+
- Linux x86-64, Windows x86-64 или macOS
- ARM-одноплатники (Raspberry Pi, Orange Pi, Rock Pi на aarch64 Linux): сборка определяет архитектуру и подтягивает ARM-нативы OpenCV / ffmpeg / PyTorch. Собирайте и запускайте прямо на плате — `./gradlew :desktop:run` — чтобы Compose/Skiko тоже взял ARM-бинарники. GPU (CUDA) на ARM недоступен, работает CPU.

### Сборка и запуск
```bash
# Запуск из исходников
./gradlew :desktop:run

# Упаковать установщик для текущей ОС
./gradlew :desktop:packageDeb      # Linux  → desktop/build/compose/binaries/main/deb/
./gradlew :desktop:packageMsi      # Windows → desktop/build/compose/binaries/main/msi/
./gradlew :desktop:packageDmg      # macOS   → desktop/build/compose/binaries/main/dmg/
```

### Использование

1. Выбери **тип модели** — ONNX или PT
2. **Browse** — укажи файл модели
3. Укажи **источник видео** — индекс веб-камеры (0 = по умолчанию) или `http://…` MJPEG URL
4. Выбери **режим GPU** — AUTO пробует CUDA, потом DirectML, потом CPU
5. Настрой размер входа, confidence, количество классов
6. Нажми **Start** — активный провайдер отображается в боковой панели
7. **MJPEG** — запустить встроенный сервер на порту 8080 для трансляции аннотированного видео
8. **Screenshot** — сохранить текущий кадр как PNG

Экспорт совместимой ONNX-модели:
```python
from ultralytics import YOLO
model = YOLO("yolov10n.pt")
model.export(format="onnx", imgsz=640, simplify=True)
```

## Одноплатник (Raspberry Pi) — headless-режим

Для запуска на плате по SSH, без рабочего стола и монитора: нет окна, приложение
просто крутит детекцию и **постоянно вещает аннотированный MJPEG-поток в локальную
сеть**. Предназначено для автозапуска при включении платы.

<details>
<summary><b>📋 Полная пошаговая инструкция с нуля (для новичков) — развернуть</b></summary>

> Инструкция написана максимально подробно, по шагам. Если что-то непонятно —
> делай ровно как написано, по порядку, не пропуская шаги. Здесь описан
> **headless-путь**: монитор, клавиатура и VNC к плате **не нужны** — ты заходишь
> на Pi по сети с обычного компьютера, а видео с рамками смотришь в браузере.

### 0. Что это вообще такое

Твой одноплатник — **Raspberry Pi 5** (мини-компьютер размером с банковскую
карту). На нём нет ни операционной системы, ни диска — их нужно записать самому на
карту памяти. Наш проект (YOLO Detector) распознаёт объекты на видео в реальном
времени и рисует вокруг них рамки. На Pi мы запускаем его **в headless-режиме**:
без окна — плата считает детекцию и раздаёт готовое видео в сеть, а ты открываешь
его в браузере на телефоне/ноутбуке.

### 1. Что нужно (одной платы и кабеля мало)

> ⚠️ На Raspberry Pi 5 порт USB-C — **только для питания**. Зайти на плату «по
> этому кабелю напрямую» нельзя; связь — по Wi-Fi или сетевому кабелю.

| Что нужно | Зачем | Подсказка |
|---|---|---|
| **microSD карта, 16 ГБ+** | Сюда ставится система | Класс 10 / A1–A2. 32 ГБ — комфортно |
| **Блок питания USB-C 5V/5A (27 Вт)** | Кабель ≠ питание, нужен зарядник | Лучше офиц. Raspberry Pi 27W. 5V/3A подойдёт, но возможны предупреждения о токе |
| **Второй компьютер + картридер** | Записать систему и заходить на плату по SSH | Любой Windows / Mac / Linux |
| **Wi-Fi или Ethernet-кабель** | Чтобы плата была в сети | Wi-Fi настроим при записи карты |
| **Источник видео** | Что распознавать | USB-вебкамера (проще всего) · **CSI-камера Pi** (ленточный шлейф) · телефон с MJPEG · SIYI-камера по RTSP |
| Монитор/VNC | **НЕ нужны** для headless | Понадобятся только если захочешь оконную GUI-версию (см. шаг 12) |

### 2. Записываем систему на карту (на втором компьютере)

1. Скачай **Raspberry Pi Imager** с `raspberrypi.com/software`, установи, вставь карту.
2. В Imager: **Choose Device** → Raspberry Pi 5; **Choose OS** → **Raspberry Pi OS
   (64-bit)** (именно 64-bit); **Choose Storage** → твоя карта (⚠️ перепроверь — всё на ней сотрётся).
3. **Next** → **Edit Settings** и заполни:
   - **Hostname**: например `yolopi`
   - **Username / Password**: придумай и **запиши** (понадобятся для входа)
   - **Configure wireless LAN**: имя своей Wi-Fi сети, пароль, страна (`RU`)
   - вкладка **Services**: галочка **Enable SSH** → **Use password authentication**
4. **Save** → **Yes / Write**. Дождись конца записи, вынь карту.

### 3. Первый запуск

1. Вставь карту в Pi (слот снизу платы).
2. Подключи питание в **USB-C**. Первая загрузка ~1–2 минуты — дай плате
   подняться и подключиться к Wi-Fi.

### 4. Заходим на плату по SSH (с второго компьютера)

```bash
ssh пользователь@yolopi.local      # например ssh pi@yolopi.local
```
При первом подключении согласись (`yes`) и введи пароль (вводится «вслепую» — это норма).

> Если `yolopi.local` не находится — глянь IP платы в роутере (вида `192.168.0.42`)
> и подключайся по нему: `ssh пользователь@192.168.0.42`

### 5. Обновляем систему и ставим зависимости

```bash
sudo apt update && sudo apt full-upgrade -y
sudo apt install -y openjdk-17-jdk git ffmpeg     # Java 17, Git, ffmpeg (HW-декод/запись)
# для CSI-камеры Raspberry Pi (если она у тебя есть):
sudo apt install -y rpicam-apps
java -version                                     # должна быть строка с 17
```

### 6. Скачиваем проект

```bash
git clone https://github.com/dest1k/yolo.git
cd yolo
```
> Поддержка ARM уже встроена — сборка сама подтянет правильные нативные библиотеки
> под процессор Pi, править ничего не нужно.

### 7. Выбираем источник видео (`YOLO_SOURCE`)

- **USB-вебкамера** → `0` (проверь, что видна: `ls /dev/video*`)
- **CSI-камера Pi** (ленточный шлейф, OV5647/IMX219) → `rpicam`
  (проверь: `rpicam-hello --list-cameras`)
- **Телефон с нашим Android-приложением** (включён MJPEG) → `http://АЙПИ-ТЕЛЕФОНА:8080`
- **Камера SIYI (ZR10 и др.)** → `rtsp://192.168.144.25:8554/main.264` (управление подвесом включится само)

### 8. Берём модель `.onnx` (на втором компьютере)

```bash
pip install ultralytics
python -c "from ultralytics import YOLO; YOLO('yolov10n.pt').export(format='onnx', imgsz=320, simplify=True)"
```
Перекинь файл на плату: `scp yolov10n.onnx пользователь@yolopi.local:~/models/`
(папку создай заранее: `mkdir -p ~/models`). Бери самую лёгкую модель — **nano**
(`yolov10n`/`yolov8n`) и вход **320**: на Pi это разумный компромисс.

### 9. Запускаем (headless)

```bash
YOLO_MODEL=~/models/yolov10n.onnx YOLO_SOURCE=0 YOLO_INPUT=320 \
  ./gradlew :desktop:runHeadless --console=plain
```
Первый запуск долгий (Gradle качает зависимости и собирает — 10–20 мин, нужен
интернет). В консоль выведется адрес вида `http://<IP-платы>:8080` — **открой его
в браузере** на телефоне/ноутбуке в той же сети. Готово: видишь видео с рамками и
счётчиком `FPS | det`.

### 10. Что можно делать в браузере

- **Панель управления** (`http://<IP-платы>:8081`): то же видео + ручной захват;
  для SIYI ещё и органы управления подвесом (для CSI/USB они скрыты, показать —
  клавиша **H**).
- **Ручной захват цели:** обведи объект мышкой (drag) — за ним поедет «приклеенная»
  рамка `LOCK`, даже если модель его не видит. Сброс — **C** или **Esc**.
- **Запись в файл** (аппаратный H.264): добавь `YOLO_RECORD=~/out.mp4` к команде
  запуска — кодирование уйдёт на аппаратный энкодёр Pi, не нагружая инференс.

Полная таблица переменных окружения — в разделе [«Конфигурация»](#конфигурация-переменные-окружения) ниже.

### 11. Автозапуск при включении (systemd)

Чтобы плата сама поднимала поток после включения — см. готовый unit-файл в разделе
[«Автозапуск при включении платы (systemd)»](#автозапуск-при-включении-платы-systemd) ниже.

### 12. (Необязательно) Оконная GUI-версия

Если очень нужна именно оконная версия (`./gradlew :desktop:run`) — ей нужен
рабочий стол: подключи монитор по **micro-HDMI** + USB-клавиатуру/мышь, либо
включи VNC (`sudo raspi-config` → Interface Options → VNC) и подключись
RealVNC Viewer'ом. Для большинства задач это **не требуется** — headless удобнее.

### 13. Чего ждать по скорости

Захват и детекция расцеплены, поэтому **видео идёт плавно** (на полном FPS камеры,
~24–30), даже если **детекция** на CPU считается медленнее — реалистично
**~10–14 кадров/с инференса** на nano-модели со входом 320. Что ускоряет:
модель **nano**, **вход 320** (или 256), и — если нужны именно MediaPipe-модели —
сайдкар (см. ниже). NPU-плат (Orange Pi 5) касается RKNN-сайдкар.

### 14. Если что-то пошло не так

- **`./gradlew: Permission denied`** → `chmod +x gradlew` и повтори.
- **Сборка падает на зависимостях** → проверь интернет: `ping -c3 github.com`.
- **`ffmpeg -i rpicam` / поток закрывается в цикле** → ты на старой версии кода;
  `git pull`. CSI идёт через `rpicam-vid`, а не через ffmpeg.
- **CSI не открывается** → `rpicam-hello --list-cameras` должна показать камеру;
  поставь `rpicam-apps`; на Pi 5 старым модулям нужен переходник шлейфа 22→15-pin.
- **Камера USB не открывается (`0`)** → `ls /dev/video*`; попробуй индекс `1`.
- **`yolopi.local` не находится** → заходи по IP платы из роутера.
- **Молния/недонапряжение на экране** → слабый блок питания, нужен 5V/5A.

### Запуск
```bash
./gradlew :desktop:runHeadless
```
GUI и дисплей не нужны. После старта в консоль выводятся адреса потока вида
`http://<IP-платы>:8080` — открой любой из них в браузере или VLC с другого
устройства в той же сети. Инференс считается только пока кто-то смотрит поток
(экономия CPU на слабой плате), сервер при этом всегда слушает порт.

Захват и детекция **расцеплены**: стрим идёт на полном FPS камеры с последними
известными рамками, а детекция крутится отдельным потоком по самому свежему
кадру (промежуточные дропаются) — это убирает инпут-лаг и не упирает видео в
скорость CPU-детектора. В левом нижнем углу — счётчик `FPS <стрим> | det <детект>`.

### Конфигурация (переменные окружения)
Всё настраивается через переменные окружения — удобно для systemd:

| Переменная | Назначение | По умолчанию |
|---|---|---|
| `YOLO_MODEL` | путь к модели `.onnx` или `.pt` | **обязательна** |
| `YOLO_MODEL_TYPE` | `onnx` или `pt` | определяется по расширению |
| `YOLO_SOURCE` | `0`/`1`… USB-камера, `rpicam` (CSI Raspberry Pi), `rtsp://…` (напр. SIYI ZR10), или `http://…` MJPEG URL | `0` |
| `YOLO_GIMBAL` | `on`/`off` — управление подвесом SIYI (авто-`on` для SIYI RTSP). Веб-панель работает всегда; для CSI/USB панель содержит только видео + ручной захват, а органы управления подвесом скрыты | `off` |
| `YOLO_GIMBAL_HOST` / `YOLO_GIMBAL_PORT` | адрес камеры SIYI (UDP SDK) | `192.168.144.25` / `37260` |
| `YOLO_CONTROL_PORT` | порт веб-панели / HTTP-API (видео + ручной захват, и подвес если есть) | `YOLO_PORT`+1 |
| `YOLO_RECORD` | путь к файлу `.mp4` — запись аннотированного видео в H.264 **аппаратным** энкодером платы (Pi `h264_v4l2m2m`, `h264_nvenc`, `h264_vaapi`…), софт `libx264` как fallback. Нужен `ffmpeg` в `PATH` | выкл |
| `YOLO_RECORD_BITRATE` | битрейт записи, кбит/с | `4000` |
| `YOLO_RECORD_ENCODER` | принудительный энкодер для записи (иначе авто-подбор) | авто |
| `YOLO_TRACK_SPEED` | макс. скорость доводки подвеса в режиме слежения | `40` |
| `YOLO_TRACK_INVERT_YAW` / `YOLO_TRACK_INVERT_PITCH` | `on` — инвертировать ось, если подвес уходит от цели | `off` |
| `YOLO_CAM_W` / `YOLO_CAM_H` / `YOLO_CAM_FPS` | разрешение и FPS захвата (для всех источников: USB и `rpicam`) | `1280` / `720` / `30` |
| `YOLO_JPEG_Q` | качество MJPEG-потока `1..100` | `80` |
| `YOLO_TRACK` | `on` / `off` — IoU-трекинг и удержание рамок | `on` |
| `YOLO_INPUT` | размер входа модели | `320` |
| `YOLO_CLASSES` | число классов | число меток из `YOLO_LABELS`, иначе `80` |
| `YOLO_LABELS` | путь к `labels.txt` (по классу на строку) для кастомных моделей | COCO |
| `YOLO_FILTER` | показывать только эти классы (индексы или имена через запятую, напр. `person` или `0,2`) — влияет на отрисовку, трекинг и слежение | все |
| `YOLO_CONF` | порог уверенности `0..1` | `0.25` |
| `YOLO_PORT` | порт MJPEG-сервера | `8080` |
| `YOLO_GPU` | `cpu` или `auto` | `cpu` |

Пример вручную:
```bash
YOLO_MODEL=/home/pi/models/yolov10n.onnx YOLO_SOURCE=0 YOLO_INPUT=320 \
  ./gradlew :desktop:runHeadless
```

**CSI-камера Raspberry Pi** (ленточный шлейф, модули на OV5647 / IMX219 и т.п.) не
видна как `/dev/videoN`, поэтому индекс `0` для неё не работает. Поставь
`YOLO_SOURCE=rpicam` — приложение возьмёт поток через `rpicam-vid`. Нужны
`rpicam-apps` (`sudo apt install -y rpicam-apps`) и работающая камера
(`rpicam-hello --list-cameras` должна её показать). На Pi 5 старым модулям нужен
переходной шлейф 22-pin → 15-pin. Для CSI/USB подвеса нет, поэтому веб-панель
(`http://<IP-платы>:8081`) показывает только видео и ручной захват цели —
органы управления ZR10 скрыты (показать/скрыть — клавиша **H**).

### Ручной захват цели (рамка мышкой)

Помимо детекций YOLO можно **захватить любой объект вручную** — даже тот, который
модель не находит. На веб-панели (`http://<IP-платы>:8081`) **обведи объект
прямоугольником мышкой** (drag) — появится намертво прикреплённая бирюзовая рамка
`LOCK`, которая следует за объектом по кадру. Это отдельный визуальный трекер
(нормированная кросс-корреляция шаблона по окну поиска, с подстройкой масштаба и
адаптацией шаблона) — лёгкий, считается на потоковом потоке и не роняет FPS на
Pi. Рамка остаётся «приклеенной» даже при кратких пропаданиях и сбрасывается сама,
если цель потеряна надолго. **Сброс захвата — клавиша C или Esc** (или эндпоинт
`/unlock`). Если подключён подвес SIYI и включено слежение (`пробел`), ручной
захват имеет приоритет: подвес ведёт именно выбранную рамку. Эндпоинты:
`/lock?x1=&y1=&x2=&y2=` (нормированные `0..1`) и `/unlock`.

### Аппаратное кодирование/декодирование

Упор на одноплатники: проект старается использовать **аппаратные блоки** платы,
а софт — только как запасной путь. Декодирование RTSP/HTTP идёт через системный
`ffmpeg` (новее и надёжнее встроенного, умеет HW-декодер платы — см.
`YOLO_HWDEC`, `YOLO_FFMPEG_HWACCEL`, `YOLO_FFMPEG_DECODER`). Запись (`YOLO_RECORD`)
автоматически берёт **аппаратный H.264-энкодер** (`h264_v4l2m2m` на Pi,
`h264_nvenc`, `h264_vaapi`, `h264_qsv`), а если его нет — `libx264 ultrafast`.
Так тяжёлое кодирование уходит с CPU, который занят инференсом. Live-видео в
браузере остаётся MJPEG (играет везде без настройки и с низкой задержкой). Пример
записи с CSI-камеры:
```bash
YOLO_MODEL=/home/pi/models/yolov10n.onnx YOLO_SOURCE=rpicam \
  YOLO_RECORD=/home/pi/out.mp4 YOLO_RECORD_BITRATE=6000 \
  ./gradlew :desktop:runHeadless --console=plain
```

### SIYI ZR10 — RTSP-поток и управление подвесом

Подключи камеру SIYI по RTSP и включи управление подвесом одной командой:
```bash
YOLO_MODEL=/home/pi/models/yolov10n.onnx \
  YOLO_SOURCE=rtsp://192.168.144.25:8554/main.264 \
  ./gradlew :desktop:runHeadless --console=plain
```
Для SIYI-адреса управление подвесом включается автоматически (или явно
`YOLO_GIMBAL=on`). В консоль выводится **единый адрес** `http://<IP-платы>:8081` —
открой в браузере и получишь **видео на весь экран с наложенными поверх
органами управления**: стрелки (движение по скорости, удержание), абсолютный
угол, центр, зум (ручной/абсолютный), фокус/автофокус, режимы lock/follow/FPV,
фото/запись/HDR и телеметрия (yaw/pitch/roll) в углу. Всё это также доступно как
HTTP-эндпоинты (`/rotate`, `/angle`, `/zoom`, `/mode`, `/status` …) — удобно
дёргать из скриптов:
```bash
curl "http://<IP>:8081/angle?yaw=30&pitch=-20"
curl "http://<IP>:8081/zoom?x=4.5"
curl "http://<IP>:8081/status"
```
Захват RTSP, отрисовка и детекция расцеплены (захват только декодирует и отдаёт
последний кадр, старые дропаются), плюс заданы low-latency опции FFmpeg. Но на
слабом CPU узкое место — **программное декодирование HEVC** 1080p/720p: если
видео всё равно отстаёт, переключи камеру на менее тяжёлый поток (более низкое
разрешение/битрейт или sub-stream, если у твоей SIYI он есть).

**Режим слежения за целью.** Нажми **пробел** на странице (или кнопку `track`) —
подвес начнёт вести найденную YOLO цель, удерживая её в центре кадра (визуальный
сервоинг по скорости, P-контроллер с мёртвой зоной; захватывается устойчивая цель
— крупнейшая/ближайшая к предыдущей, после короткой стабилизации). **Клик мышкой
по объекту** на видео мгновенно захватывает именно его (и включает слежение).
Включённый режим виден бейджем `● TRACK ON` сверху и перекрестием + жёлтой рамкой
на цели в видео. Скорость доводки — `YOLO_TRACK_SPEED`; если подвес уходит от цели, а не к
ней — инвертируй ось через `YOLO_TRACK_INVERT_YAW`/`_PITCH`. Эндпоинт:
`/track` (тумблер) или `/track?on=1`.

Протокол — UDP SIYI SDK (камера `192.168.144.25:37260`); поддержаны зум, фокус,
вращение по скорости, абсолютный угол, центрирование, режимы движения,
фото/запись/HDR и запросы версии/ID/аттитюда/конфига. Реализовано по
опубликованному SDK — проверь на своём ZR10, при расхождениях на уровне байт
поправлю.

### Автозапуск при включении платы (systemd)

1. Создай файл сервиса `/etc/systemd/system/yolo-detector.service` (подставь свои
   путь к проекту, модель и имя пользователя):

   ```ini
   [Unit]
   Description=YOLO Detector (headless MJPEG broadcast)
   After=network-online.target
   Wants=network-online.target

   [Service]
   Type=simple
   User=pi
   WorkingDirectory=/home/pi/yolo
   Environment=YOLO_MODEL=/home/pi/models/yolov10n.onnx
   Environment=YOLO_SOURCE=0
   Environment=YOLO_INPUT=320
   Environment=YOLO_PORT=8080
   Environment=YOLO_GPU=cpu
   ExecStart=/home/pi/yolo/gradlew :desktop:runHeadless --console=plain
   Restart=on-failure
   RestartSec=5

   [Install]
   WantedBy=multi-user.target
   ```

2. Включи и запусти сервис:
   ```bash
   sudo systemctl daemon-reload
   sudo systemctl enable --now yolo-detector
   ```

3. Полезные команды:
   ```bash
   systemctl status yolo-detector      # состояние
   journalctl -u yolo-detector -f      # живой лог (адреса потока — здесь)
   sudo systemctl restart yolo-detector
   ```

> Первый запуск дольше: Gradle собирает проект и тянет зависимости (нужен
> интернет). Чтобы плата не собирала проект при каждом старте — это уже учтено:
> Gradle кеширует сборку, последующие запуски быстрые. Поток вещается на CPU, так
> что на лёгкой модели (yolov10n, вход 320) жди несколько кадров в секунду.

### Orange Pi 5 / RK3588 (NPU, RKNN)

На платах Rockchip с NPU (Orange Pi 5 и др.) CPU-инференс — не лучший выбор: есть
NPU на ~6 TOPS. Для него в репозитории лежит **Python-сайдкар** на NPU
(`tools/rknn-sidecar/`): он гоняет `.rknn`-модель на NPU и раздаёт такой же
аннотированный MJPEG-поток (с FPS-счётчиком и трекингом), как JVM-версия. RKNN
управляется из Python, поскольку у его рантайма нет Java-биндинга. Конвертация
модели ONNX → `.rknn` и запуск описаны в [`tools/rknn-sidecar/README.md`](tools/rknn-sidecar/README.md).

### Raspberry Pi 5 — MediaPipe Object Detection (CPU/XNNPACK)

Если хочется использовать модели **MediaPipe Object Detection** (EfficientDet-Lite
или свои из MediaPipe Model Maker), в репозитории есть **Python-сайдкар**
(`tools/mediapipe-sidecar/`). Он гоняет `.tflite`-модель через MediaPipe Tasks и
раздаёт такой же аннотированный MJPEG-поток и веб-панель, как JVM-версия —
**включая ручной захват цели мышкой** и управление подвесом SIYI. На Pi 5
MediaPipe считает на CPU через делегат **XNNPACK** (4 ядра A76) — это и есть
рабочий «ускоренный» путь на Pi (пригодного GPU/NPU-делегата для VideoCore нет).
Конвертировать ничего не нужно — указываешь `.tflite` и запускаешь. Установка,
скачивание модели и запуск (в т.ч. CSI через `rpicam`) — в
[`tools/mediapipe-sidecar/README.md`](tools/mediapipe-sidecar/README.md).

```bash
pip3 install mediapipe python3-opencv   # см. README сайдкара
YOLO_MODEL=~/models/efficientdet_lite0.tflite YOLO_SOURCE=rpicam \
  python3 tools/mediapipe-sidecar/yolo_mediapipe_sidecar.py
# затем открой http://<IP-платы>:8080  (обведи объект рамкой, чтобы захватить)
```

### Raspberry Pi 5 — PicoDet (NCNN, быстрый CPU-инференс)

**PicoDet** (PaddleDetection) — один из самых быстрых детекторов на ARM-CPU. Для
него есть **Python-сайдкар на NCNN** (`tools/picodet-sidecar/`): раздаёт такой же
аннотированный MJPEG-поток и панель (с ручным захватом и подвесом), но декодирует
GFL-голову PicoDet сам (NCNN не умеет встроенный NMS PicoDet). В комплекте —
конвертер датасета в COCO (`dataset_to_coco.py`) и обёртка для обучения через
PaddleDetection (`train_picodet.py`). Имена выходных блобов/`cell_offset` зависят
от экспорта, поэтому при первом запуске используй `--inspect` — подробно в
[`tools/picodet-sidecar/README.md`](tools/picodet-sidecar/README.md).

```bash
pip3 install ncnn numpy python3-opencv
PICODET_PARAM=picodet.param PICODET_BIN=picodet.bin python3 \
  tools/picodet-sidecar/picodet_ncnn_sidecar.py --inspect   # узнать имена блобов
```

> ⚠️ Обучение PicoDet идёт через **PaddleDetection** (тяжело, ~300 эпох) и требует
> поддерживаемого NVIDIA GPU — Blackwell (RTX 50xx) не подойдёт и тут, нужен
> Colab/облако. Инференс на Pi от этого не зависит.

### Raspberry Pi 5 — YOLO-FastestV2 (NCNN, максимальный FPS)

**YOLO-FastestV2** (dog-qiuqiu) — один из самых маленьких детекторов (~0.25M
параметров), на голом CPU Pi 5 обычно даёт **наибольший FPS** из всех вариантов.
Сайдкар — `tools/yolo-fastestv2-sidecar/` (NCNN, та же панель с ручным захватом и
подвесом). Плюсы для обучения: датасет — **прямо YOLO-формат** (конвертить боксы не
надо), а тренировка — **обычный PyTorch**, который на свежей сборке (cu128)
**работает и на Blackwell** — то есть твою RTX 5090 наконец можно задействовать. В
комплекте: подготовка данных (`make_yolofastest_data.py`) и **обучение одной
командой** `train_yolofastest.py` — оно само строит `.data`, клонирует репозиторий,
считает якоря, тренирует и **автоматически экспортирует оптимизированную ncnn-модель
с проверкой** (`export_ncnn.py`: opset 11 + `dynamo=False` + onnxsim → ncnn →
ncnnoptimize fp16 → загрузка-проверка). Готовый `.param`/`.bin` работает и в сайдкаре,
и в Android-приложении (версия = FastestV2). Подробно в
[`tools/yolo-fastestv2-sidecar/README.md`](tools/yolo-fastestv2-sidecar/README.md).

```bash
# готовую COCO-модель для первого теста (до обучения) скачает скрипт:
python3 tools/yolo-fastestv2-sidecar/get_model.py
# он печатает готовую команду запуска. То же есть в каждой папке сайдкара
# (mediapipe — официальный tflite; picodet — конвертит из Paddle; nanodet —
#  конвертит чекпойнт; rknn — это для Rockchip, не для Pi 5).
```

Нужна **точность по мелким объектам** (а не только максимальный FPS) — есть
**NanoDet-Plus** (anchor-free, FPN 8/16/32/64): тоже обучается на Blackwell одной
командой и экспортируется в ncnn. Сайдкар — `tools/nanodet-sidecar/`, подробно в
[`tools/nanodet-sidecar/README.md`](tools/nanodet-sidecar/README.md). Обе модели
работают на Pi 5, в desktop-headless (`YOLO_DECODE=nanodet` для NanoDet) и на телефоне.

> 📘 **Пошаговый мануал от датасета до рабочей модели** (обучение на RTX 5090 →
> деплой на Pi 5 / headless / телефон), включая **графическую обучалку** и
> **генератор синтетического датасета** — ниже:
> [Обучение своих моделей](#обучение-своих-моделей--от-датасета-до-pi-5).

## Форматы моделей и конвертеры

Каждый рантайм хочет свой формат. Сводка «откуда что взять»:

| Куда (рантайм) | Формат | Как получить из стоковой модели |
|---|---|---|
| **Desktop (JVM)** | `.onnx` / `.pt` | `YOLO('yolov10n.pt').export(format='onnx', imgsz=320, simplify=True)` (Ultralytics). `.pt` (TorchScript) — тоже грузится |
| **RKNN-сайдкар** (Orange Pi 5 / NPU) | `.rknn` | `.onnx` → **rknn-toolkit2** на x86-хосте (см. [rknn README](tools/rknn-sidecar/README.md)) |
| **MediaPipe-сайдкар** (Pi 5) | `.tflite` | **конвертировать не надо** — скачать готовую EfficientDet-Lite или обучить через Model Maker ([README](tools/mediapipe-sidecar/README.md)) |
| **PicoDet-сайдкар** (Pi 5, NCNN) | `.param` + `.bin` | PaddleDetection → ONNX (**без NMS**) → `onnxsim` → `onnx2ncnn` ([README](tools/picodet-sidecar/README.md)) |
| **YOLO-FastestV2-сайдкар** (Pi 5, NCNN) | `.param` + `.bin` | репо → ONNX → `onnxsim` → `onnx2ncnn` ([README](tools/yolo-fastestv2-sidecar/README.md)) |

Базовые инструменты конвертации:
```bash
# PT → ONNX (Ultralytics YOLO)
pip install ultralytics
python -c "from ultralytics import YOLO; YOLO('yolov10n.pt').export(format='onnx', imgsz=320, simplify=True)"

# ONNX → упростить (всегда полезно перед onnx2ncnn)
pip install onnxsim && python -m onnxsim model.onnx model-sim.onnx

# ONNX → NCNN (.param/.bin) — бинарь onnx2ncnn из сборки инструментов ncnn
onnx2ncnn model-sim.onnx model.param model.bin
# ВАЖНО: onnx2ncnn ждёт opset 11. На opset 13/17/18 он криво конвертит shape-операции —
# модель грузится, но падает на forward (ncnn -100). Экспортируй ONNX с opset 11
# (и прогоняй onnxsim), либо для новых opset используй pnnx вместо onnx2ncnn.

# ONNX → RKNN — на x86 Linux, см. rknn-sidecar/README.md
# (PT/ONNX → TFLite для MediaPipe обычно не нужен — там свой Model Maker)
```

Подробные пошаговые рецепты (включая нюансы экспорта, например PicoDet **без NMS**
для NCNN) — в README соответствующего сайдкара. NCNN-сайдкары при первом запуске
сами определяют структуру модели (`auto: …`), а формулу декода можно подобрать
флагом `--autotune` — конвертация остаётся единственным ручным шагом.

## Обучение своих моделей — от датасета до Pi 5

Парадигма одна для всех детекторов: **правишь `CONFIG`-блок (или жмёшь кнопки в
графической обучалке) и запускаешь один скрипт** — он сам соберёт данные, склонирует
upstream-репозиторий, обучит и **экспортирует проверенную ncnn-модель**. Обучение —
на Windows + NVIDIA GPU (включая Blackwell / RTX 50xx на сборке cu128 nightly),
инференс — на Raspberry Pi 5 (CPU). Та же `.param`/`.bin` работает и в desktop-headless,
и в Android-приложении.

### 0. Окружение на машине обучения (Windows, в venv)
```bash
python -m venv yolo-venv
yolo-venv\Scripts\activate
# PyTorch для Blackwell (RTX 50xx) — обязательно сборка cu128:
pip install --pre torch torchvision --index-url https://download.pytorch.org/whl/nightly/cu128
# общие инструменты экспорта/проверки:
pip install opencv-python numpy onnx onnxsim ncnn pnnx
# NanoDet дополнительно:
pip install pytorch-lightning pycocotools omegaconf tensorboard
```
Проверь GPU: `python -c "import torch; print(torch.cuda.is_available())"` → `True`.
Если `False` — стоит не та сборка torch (нужна **cu128 nightly**).

### 1. Датасет (YOLO-формат)
```
<dataset>/
  images/train/*.jpg   images/val/*.jpg
  labels/train/*.txt   labels/val/*.txt
```
Каждый `.txt` — по строке на объект: `class_id  xc yc w h` (нормировано 0..1).
Имена классов задаются списком в порядке id. Нужен и `val` (10–20% данных). Нет
своего датасета? Сгенерируй синтетику — см. [генератор](#генератор-синтетического-датасета-flux--llm--авторазметка).

### 2. Выбор модели
| | Когда брать | FPS @Pi5 | Мелкие объекты | Где работает |
|---|---|---|---|---|
| **YOLO-FastestV2** | нужен максимальный FPS | ★★★ (~78 @320) | ★ | Pi + headless + телефон |
| **NanoDet-Plus** | нужна точность по мелочи | ★★ (≥30) | ★★★ | Pi + headless + телефон |

Рекомендация: начни с **FastestV2**; теряет мелкие объекты — переходи на **NanoDet-Plus**
(FPN, страйд-8). Обе обучаются на Blackwell и экспортируются в ncnn одной командой.
Проверить сайдкар ещё до обучения можно готовой COCO-моделью: `python tools/<sidecar>/get_model.py`.

### 3. Запуск обучения

#### Графическая обучалка (NanoDet, Windows)
Не хочешь править файлы — открой окно:
```bash
cd tools/nanodet-sidecar
python train_nanodet_gui.py
```
В окне задаются **все** параметры (датасет с кнопкой «Browse», классы, размер входа,
эпохи, батч, воркеры, устройство, GPU ids, learning rate, reg_max, авто-экспорт), есть
отдельный блок **дообучения**, а **снизу — живой лог** всего процесса (конвертация
данных → обучение → экспорт → `VERIFY OK`). Под капотом запускается тот же
`train_nanodet.py`, поля просто прокидываются как переменные `TRAIN_*` — поэтому
консольный и оконный пути идентичны, а значения полей запоминаются между запусками.

Нужен только Python с **tkinter** (на Windows идёт из коробки; Linux —
`sudo apt install python3-tk`). torch/nanodet нужны самой обучалке, а не окну.

#### Из консоли
Открой `train_yolofastest.py` или `train_nanodet.py`, в `CONFIG`-блоке задай
`DATASET`, `CLASSES`, `INPUT`, `EPOCHS` (железо уже выставлено под 285K/5090), и:
```bash
cd tools/yolo-fastestv2-sidecar && python train_yolofastest.py   # FastestV2
cd tools/nanodet-sidecar        && python train_nanodet.py       # NanoDet-Plus
```
Скрипт: `[1/4]` готовит данные · `[2/4]` клонирует репозиторий · `[3/4]` обучает ·
`[4/4]` **экспортирует и проверяет** ncnn-модель.

#### Дообучение (continue-training / дообучение)
Чтобы продолжить уже обученную модель на новых/расширенных данных — укажи путь к
готовому `.ckpt`:
- **`WEIGHTS`** (env `TRAIN_WEIGHTS`) — стартовать с этих весов и доучиться на твоих
  данных (transfer learning). Это и есть обычное «дообучение».
- **`RESUME`** (env `TRAIN_RESUME`) — продолжить **прерванный** запуск с сохранением
  оптимизатора и счётчика эпох.

В графической обучалке это переключатель **Fine-tuning / resume** с выбором чекпойнта.

### 4. Результат экспорта
В конце увидишь **`VERIFY OK`** и пути к файлам: `*.param` + `*.bin` (оптимизированная
fp16 ncnn-модель) и `classes.txt`/`custom.names`. `VERIFY OK` = модель реально
грузится и блобы извлекаются — можно нести на Pi. **Инференс должен идти на том же
`INPUT`, что и обучение** (иначе боксы поедут).

### 5. Запуск на Raspberry Pi 5
Скопируй `*.param`, `*.bin` и файл имён классов на Pi, поставь рантайм
`pip3 install ncnn numpy opencv-python` и запусти сайдкар:
```bash
# NanoDet-Plus:
ND_PARAM=nanodet.param ND_BIN=nanodet.bin ND_INPUT=416 \
  YOLO_LABELS=classes.txt YOLO_SOURCE=rpicam \
  python3 tools/nanodet-sidecar/nanodet_ncnn_sidecar.py
# YOLO-FastestV2:
YF_PARAM=yolofastestv2.param YF_BIN=yolofastestv2.bin YF_INPUT=416 \
  YOLO_LABELS=custom.names YOLO_SOURCE=rpicam \
  python3 tools/yolo-fastestv2-sidecar/yolofastest_ncnn_sidecar.py
```
Открой `http://<ip-пи>:8080`. Боксов нет / кривые — сначала `--inspect` (перебирает
размеры входа и печатает форму выхода; чаще всего `*_INPUT` ≠ размеру экспорта).

### 6. Desktop-headless и телефон
**Headless** (JVM, ONNX/PT) — берёт промежуточный `*-sim.onnx` из экспорта; для NanoDet
укажи `YOLO_DECODE=nanodet ND_REG_MAX=7`. **Телефон**: перекинь `*.param`+`*.bin`,
«Определить выходы», выбери версию (FastestV2/NanoDet), размер входа = `INPUT`, число
классов под модель.

### Быстродействие на Pi 5
- модель **nano**, вход **320/256** — главный рычаг;
- ncnn-сайдкары теперь **по умолчанию включают fp16-арифметику + winograd/sgemm**
  ядра (Cortex-A76 = ARMv8.2 FP16) — заметный прирост FPS без потери качества;
  откатиться на fp32 — `YOLO_FP16=0`;
- `*_THREADS=4` (по числу ядер A76), `YOLO_CV_THREADS=1` — чтобы OpenCV не отбирал
  ядра у инференса;
- захват и детекция расцеплены — видео идёт на полном FPS камеры даже когда детектор
  считает медленнее; запись — на **аппаратный H.264-энкодер** платы (`YOLO_RECORD`).

### Генератор синтетического датасета (FLUX + LLM + авторазметка)
Нет своих данных — собери их сам: **`tools/dataset-generator/`** генерирует картинки
через **FLUX.1-schnell**, промпты к ним пишет локальная **LLM** (LM Studio / любой
OpenAI-совместимый эндпоинт), а боксы расставляет **YOLO-World** (zero-shot) — на
выходе готовый YOLO-датасет с `dataset.yaml`. Три фазы (промпты → картинки →
разметка) включаются по отдельности, длинный прогон стримится на диск и
возобновляется.

**Всё управляется из окна** — `python tools/dataset-generator/generate_dataset_gui.py`:
вкладки Run / LLM / Image / **Prompts** / Labeling и живой лог снизу. На вкладке
**Prompts** — полный редактор словаря промптов: типы дронов, материалы, фоны, погода,
состояния, ракурсы, **микс масштабов** (вес + фраза, дефицит-планировщик держит
пропорции точно) и сырой шаблон system-prompt. Окно пишет JSON-конфиг и запускает
движок `generate_dataset.py` (его же можно гонять headless: `python generate_dataset.py config.json`).
Подробности и требования — в [`tools/dataset-generator/README.md`](tools/dataset-generator/README.md).

## Follow-me: дрон следит за человеком (Betaflight FC)

Механика **lock-on теперь работает и без подвеса** — с простой RPi-камерой. Жмёшь
**пробел** (с ноутбука, на веб-панели) — сайдкар захватывает цель и держит
перекрестие. **Целей несколько?** Кликни мышкой по нужному человеку — захват
переключится именно на него (а перетаскиванием рамки можно «прибить» вообще любой
объект, даже тот, что модель не видит). Работает во всех режимах: простая камера,
подвес или дрон. Если к Raspberry Pi подключён
**полётный контроллер на Betaflight**, этот же захват **управляет дроном**: дрон
доворачивает по **yaw**, удерживая человека в центре кадра, и **подлетает/отлетает**
(pitch), удерживая дистанцию — размер бокса служит датчиком расстояния: человек
уходит (бокс уменьшается) → дрон летит вперёд; приближается (бокс растёт) → отходит.

Реализовано в NanoDet-сайдкаре (`tools/nanodet-sidecar/`, файл `flight_controller.py`).
Управление идёт по **MSP** (`MSP_SET_RAW_RC`) поверх **MSP Override** — Pi берёт только
yaw+pitch, а **газ, арм и failsafe остаются у пилота на пульте**.

> ⚠️ **БЕЗОПАСНОСТЬ — прочитай до подключения дрона.** Это реальный полёт в сторону
> человека. **Все первые проверки — со снятыми пропеллерами.** Летай только на
> открытой площадке, вдали от людей, под свою ответственность и по местным правилам.
> **Два предохранителя:** аппаратный — **MSP Override** на тумблере AUX пульта (сбросил
> тумблер → мгновенно вернул себе управление), и программный — кнопка **ARM** на
> веб-панели (пока OFF — Pi не шлёт на дрон ничего). Держи пульт включённым.
> При потере цели или выключенном слежении
> сайдкар шлёт **центр стиков** (дрон висит, не «уезжает»); если процесс на Pi умрёт —
> поток MSP прекратится и сработает failsafe полётника.

### Как это работает простыми словами

У дрона уже есть мозг (полётный контроллер на Betaflight) и пульт. Мы добавляем
**второй мозг — Raspberry Pi с камерой**. Pi смотрит в камеру, ловит человека нейросетью
и говорит полётнику по проводу «крути нос влево/вправо, лети вперёд/назад» — ровно так,
как будто это делает пилот стиками. Но Pi разрешено трогать **только две ручки**
(поворот и вперёд-назад) и **только когда ты включил тумблер** на пульте. Газ, взлёт,
посадку и аварийный сброс всегда держишь ты.

### Что понадобится (список покупок)

- **Дрон с полётником на Betaflight** и обычным приёмником (CRSF/ELRS/SBUS) — пульт остаётся.
- **Raspberry Pi 5** + камера: USB-вебка или CSI-камера Pi (`YOLO_SOURCE=rpicam`).
- **Отдельное питание для Pi 5** (повербанк/BEC 5 В **5 А**) — питать Pi 5 от полётника плохо.
- **Кабель для связи Pi↔полётник**:
  - для **USB-варианта** — тот же кабель, которым ты подключаешь полётник к Betaflight
    Configurator на компе (USB-A на стороне Pi);
  - для **UART-варианта** — 3 тонких провода и паяльник.
- На Pi: `pip3 install pyserial ncnn numpy opencv-python`.
- **COCO-модель детектора** (чтобы вообще был класс «person»). Если у тебя своя модель
  только на дроны — она людей не увидит. Возьми COCO-модель и добавь `YOLO_FILTER=person`.

### Выбери свой способ подключения

| | USB (без пайки) | UART (пайка) |
|---|---|---|
| Сложность | проще некуда | нужен паяльник |
| Надёжность в полёте | разъём может выскочить от вибрации | припаяно намертво |
| Проверка направлений | Configurator занят (порт один) | можно держать Configurator открытым |
| **Кому** | **попробовать/настроить на столе** | **боевой полёт** |

Ниже — **две независимые инструкции**. Открой ту, что под твой способ подключения, и
пройди её **целиком сверху вниз** — каждая самодостаточна (общие шаги продублированы),
прыгать между ними не нужно. Совет новичку: начни с USB на столе.

<details>
<summary><b>🔌 ИНСТРУКЦИЯ 1 — подключение по USB (без пайки). Развернуть полностью</b></summary>

> Здесь Pi и полётник соединяются обычным USB-кабелем. Полётник по USB всегда
> отдаёт MSP — это тот же канал, по которому с ним говорит Betaflight Configurator.

#### A1. Соедини
1. Возьми USB-кабель полётника (которым подключаешь его к компу).
2. Воткни его одним концом в **полётник**, другим — в **любой USB-порт Raspberry Pi**.
3. Питание: **полётник** в полёте питается от **аккумулятора дрона** как обычно (USB —
   это только данные). **Raspberry Pi** питай от **своего** источника (повербанк/BEC 5 В 5 А).
   «Землю» они разделят через USB автоматически — отдельный провод GND не нужен.

> ⚠️ На столе для настройки можно запитать полётник и от самого USB (моторы при этом
> не крутятся — это нормально). Для полёта полётнику нужен аккумулятор.

#### A2. Найди порт на Pi
В терминале Pi:
```bash
ls /dev/ttyACM*        # обычно появится /dev/ttyACM0
```
Не появилось — выдерни и снова воткни кабель и смотри, что добавилось:
```bash
dmesg | tail -n 20     # ищи строку вида "ttyACM0: USB ACM device"
```
Запомни путь (чаще всего `/dev/ttyACM0`). Это и будет `FC_PORT`.

Дай себе права на порт (один раз), чтобы не запускать через sudo:
```bash
sudo usermod -aG dialout $USER     # затем выйди из сессии и зайди снова (или перезагрузись)
```

#### A3. Настрой Betaflight (с компа, один раз)
Подключи полётник к **компьютеру** и открой Betaflight Configurator:
1. **Ports**: для USB **ничего менять не надо** — VCP уже отдаёт MSP. Приёмник свой не трогай.
2. **Modes** → найди строку **MSP OVERRIDE** → **Add Range** → выбери свободный **AUX**
   (тумблер на пульте) и задай диапазон, где тумблер «включён» (напр. 1700–2100). Save.
3. **CLI** (вкладка CLI) — разреши Pi трогать только повороты и вперёд-назад
   (roll+pitch+yaw, газ/арм НЕ отдаём):
   ```
   set msp_override_channels_mask = 11
   save
   ```
   Откуда 11: бит0=roll(1) + бит1=pitch(2) + бит3=yaw(8) = 11. (В старых прошивках
   параметр зовётся `msp_override_channels`.)
4. **Failsafe**: проверь, что настроен (Stage 2 = Drop/Land).
5. **Отключи Configurator от полётника** — иначе Pi не сможет занять тот же USB-порт
   (MSP-клиент может быть только один).

#### A4. Запусти на Pi
```bash
pip3 install pyserial ncnn numpy opencv-python
ND_PARAM=nanodet.param ND_BIN=nanodet.bin ND_INPUT=320 \
  YOLO_LABELS=coco.names YOLO_FILTER=person YOLO_SOURCE=rpicam \
  FC=betaflight FC_PORT=/dev/ttyACM0 \
  python3 tools/nanodet-sidecar/nanodet_ncnn_sidecar.py
```
В консоли появится жёлтая плашка «BETAFLIGHT FOLLOW MODE ENABLED» и адрес
`http://<ip-пи>:8080`.

> Особенность USB: пока Pi держит порт, Betaflight Configurator к этому полётнику по
> USB одновременно не подключишь (MSP-клиент один). Поэтому направления ниже
> проверяем по реакции моторов со снятыми пропами.

#### A5. Проверка БЕЗ пропеллеров (обязательно)
> 🚨 **Сними пропеллеры.** Здесь убеждаемся, что дрон поворачивается в **правильную**
> сторону. Перепутанный знак = поедет ОТ человека / в стену.
1. Открой `http://<ip-пи>:8080` с ноутбука/телефона в той же Wi-Fi сети.
2. Подай питание на дрон (**пропы сняты!**), включи пульт, **заармь** как обычно.
3. Включи на пульте тумблер **MSP Override**.
4. На веб-панели нажми красную кнопку **ARM** (это **программный предохранитель** — пока
   он OFF, Pi не отправляет на дрон ничего: ни авто-слежение, ни ручные команды).
5. Нажми **пробел** (бейдж станет «LOCK ON») и пройди перед камерой — на тебе появится
   рамка. Людей несколько — **кликни мышкой по себе**.
6. Смотри по моторам (пропы сняты!): человек **справа** в кадре → дрон должен пытаться
   повернуть **вправо**; человек **далеко/мелкий** → тяга **вперёд**. Не туда —
   перезапусти Pi, добавив `FC_INVERT_YAW=1` и/или `FC_INVERT_PITCH=1`.
7. Выключи **ARM** на панели, сбрось тумблер **MSP Override**, **разармь**, выключи питание.

#### A6. Что дальше — полёт
На USB-привязи к Pi это путь «для стола»: ты подключился, настроил Betaflight и
проверил направления. **Для реального полёта перенеси связь на UART** (надёжнее при
вибрации, и видно каналы в Configurator) — там же, в **Инструкции 2**, шаги первого
полёта и полная таблица тонкой настройки `FC_*` (а также в
[README сайдкара](tools/nanodet-sidecar/README.md#follow-me-flight-betaflight-fc--lock-on-without-a-gimbal)).

#### A7. Если что-то не так
| Симптом | Причина / что делать |
|---|---|
| `pyserial not installed` | `pip3 install pyserial` на Pi |
| `cannot open FC serial port` | неверный `FC_PORT` (`ls /dev/ttyACM*`), нет прав (`usermod -aG dialout`, перелогиниться), кабель |
| Pi не видит порт | другой кабель/порт Pi; `dmesg \| tail`; полётник запитан? |
| Дрон не реагирует | не включён тумблер **MSP Override**; маска `msp_override_channels_mask` не сохранена |
| Крутит/едет не в ту сторону | `FC_INVERT_YAW=1` и/или `FC_INVERT_PITCH=1` (сначала **со снятыми пропами!**) |
| Configurator не подключается, пока работает Pi | по USB MSP-клиент один: закрой Pi (или используй UART) |
| Человека не видит | нужна **COCO-модель** + `YOLO_FILTER=person` (своя «дроновая» модель людей не знает) |

</details>

<details>
<summary><b>🔧 ИНСТРУКЦИЯ 2 — подключение по UART (пайка). Развернуть полностью</b></summary>

> Здесь 3 провода паяются между Pi и свободным UART полётника. Надёжно для полёта и
> позволяет одновременно держать открытым Betaflight Configurator по USB.

#### B1. Что куда паять
Логика у Pi и полётника **3.3 В** — совместимы напрямую. Провода **TX↔RX
крест-накрест** + общий **GND**. **Сигнальные 5 В между платами не соединяй.**

Найди на полётнике **свободный UART** (площадки с подписью вроде `T2`/`R2` = TX/RX
второго UART; занятый приёмником/GPS не бери).

| Raspberry Pi 5 (40-пиновый разъём) | Физический пин | → | Полётник (свободный UART) |
|---|---|---|---|
| GPIO14 / **TXD0** (передача) | пин **8** | → | **RX** этого UART |
| GPIO15 / **RXD0** (приём) | пин **10** | → | **TX** этого UART |
| **GND** (земля) | пин **6** | → | **GND** |

Нумерация пинов Pi: смотри на плату USB-портами вниз, левый столбец сверху — пин 1.
Пин 6 — третий сверху в правом столбце (GND), пины 8 и 10 — четвёртый и пятый справа.

> Питание Pi 5 — **отдельным БП 5 В/5 А**. Если очень надо от платы — только мощный
> 5 В BEC на пин **2 или 4 (5V)** Pi + общий GND. От USB-порта полётника Pi 5 не питай.

#### B2. Включи UART на Pi
```bash
sudo raspi-config
```
→ **Interface Options** → **Serial Port** →
«Would you like a login shell over serial?» → **No** →
«Would you like the serial port hardware enabled?» → **Yes** → Finish → **перезагрузка**.

Порт будет `/dev/ttyAMA0` (он же `/dev/serial0`). Это `FC_PORT`. Проверка:
```bash
ls -l /dev/serial0      # должен ссылаться на ttyAMA0
```

#### B3. Настрой Betaflight (с компа, один раз)
1. **Ports** → на том UART, что распаял, включи **MSP** и поставь **115200**. Save & Reboot.
   Приёмник свой **не** переводи в MSP — оставь CRSF/ELRS/SBUS.
2. **Modes** → **MSP OVERRIDE** → **Add Range** → свободный **AUX** (тумблер), диапазон
   «включено» (напр. 1700–2100). Save.
3. **CLI**:
   ```
   set msp_override_channels_mask = 11
   save
   ```
   (roll+pitch+yaw = 1+2+8 = 11; в старых прошивках — `msp_override_channels`.)
4. **Failsafe**: проверь (Stage 2 = Drop/Land).

#### B4. Запусти на Pi
```bash
pip3 install pyserial ncnn numpy opencv-python
ND_PARAM=nanodet.param ND_BIN=nanodet.bin ND_INPUT=320 \
  YOLO_LABELS=coco.names YOLO_FILTER=person YOLO_SOURCE=rpicam \
  FC=betaflight FC_PORT=/dev/ttyAMA0 FC_BAUD=115200 \
  python3 tools/nanodet-sidecar/nanodet_ncnn_sidecar.py
```
В консоли появится жёлтая плашка «BETAFLIGHT FOLLOW MODE ENABLED» и адрес
`http://<ip-пи>:8080`. По UART можно **одновременно** держать Betaflight Configurator
открытым по USB — это удобно для проверки направлений.

#### B5. Проверка БЕЗ пропеллеров (обязательно)
> 🚨 **Сними пропеллеры.** Здесь убеждаемся, что дрон поворачивается в **правильную**
> сторону. Перепутанный знак = поедет ОТ человека / в стену.
1. Открой `http://<ip-пи>:8080` с ноутбука/телефона в той же Wi-Fi сети. По желанию
   подключи полётник к компу по USB и открой **Configurator → вкладка Receiver**.
2. Подай питание на дрон (**пропы сняты!**), включи пульт, **заармь** как обычно.
3. Включи на пульте тумблер **MSP Override**.
4. На веб-панели нажми красную кнопку **ARM** (это **программный предохранитель** — пока
   он OFF, Pi не отправляет на дрон ничего: ни авто-слежение, ни ручные команды).
5. Нажми **пробел** (бейдж станет «LOCK ON») и пройди перед камерой — на тебе появится
   рамка. Людей несколько — **кликни мышкой по себе**.
6. Смотри в Receiver-вкладке (или по моторам): человек **справа** в кадре → канал
   **Yaw** должен идти в сторону поворота **вправо**; человек **далеко/мелкий** → канал
   **Pitch** в сторону **вперёд**. Не туда — перезапусти Pi, добавив `FC_INVERT_YAW=1`
   и/или `FC_INVERT_PITCH=1`.
7. Выключи **ARM** на панели, сбрось тумблер **MSP Override**, **разармь**, выключи питание.

#### B6. Первый полёт
Только когда направления верны: открытая площадка, **никого вокруг**, дрон в режиме
**Angle/Horizon**. Взлети и держи **газ сам**. Включи **MSP Override**, на панели нажми
**ARM**, затем **пробел** и кликни нужного человека — дрон начнёт доворачивать и держать
дистанцию. **Чуть что — выключи ARM на панели или сбрось тумблер** (управление мгновенно
вернётся к тебе). Начинай с малых `FC_MAX_YAW`/`FC_MAX_PITCH`.

**Ручное управление с панели.** Когда подключён FC, на панели появляется **d-pad**:
▲/▼ — вперёд/назад, ⟲/⟳ — поворот (yaw), ◄/► — боком (roll). Работает **только при ARM**;
кнопку держишь — летит, отпустил — стики в центр (и если страница пропала — тоже центр
через ~0.4 с). Ручная команда на пару мгновений главнее авто-слежения.

#### B7. Тонкая настройка (переменные окружения)
Добавляй их в ту же команду запуска (`B4`):

| Переменная | Что делает | По умолчанию |
|---|---|---|
| `FC_TARGET_FILL` | какую долю высоты кадра должен занимать человек (дистанция). Больше = ближе | `0.45` |
| `FC_MAX_YAW` / `FC_MAX_PITCH` | максимальный «отклон ручки» от центра, мкс (резкость) | `150` / `120` |
| `FC_KP_YAW` / `FC_KP_PITCH` | усиление: насколько резко реагировать на ошибку | `360` / `500` |
| `FC_YAW_DEADZONE` / `FC_FILL_DEADZONE` | мёртвая зона (мелкие отклонения игнорируются) | `0.06` / `0.08` |
| `FC_INVERT_YAW` / `FC_INVERT_PITCH` | перевернуть ось, если едет не туда | off |
| `FC_CH_ROLL/PITCH/THROTTLE/YAW` | индексы каналов, если `map` в Betaflight не AETR | `0/1/2/3` |
| `FC_BAUD` / `FC_RATE` | скорость UART / частота отправки RC, Гц | `115200` / `50` |

#### B8. Если что-то не так

| `cannot open FC serial port` | неверный `FC_PORT` (`/dev/serial0`→`ttyAMA0`), serial не включён в `raspi-config`, провода |
| Дрон не реагирует | не включён тумблер **MSP Override**; в **Ports** не включён MSP на этом UART; маска `msp_override_channels_mask` не сохранена |

| Крутит/едет не в ту сторону | `FC_INVERT_YAW=1` и/или `FC_INVERT_PITCH=1` (сначала **со снятыми пропами!**) |
| Дёргается/перелетает цель | уменьши `FC_KP_YAW`/`FC_KP_PITCH` и `FC_MAX_*`, увеличь мёртвые зоны |
| Человека не видит | нужна **COCO-модель** + `YOLO_FILTER=person` (своя «дроновая» модель людей не знает) |

### Управление стиками с пульта (RadioMaster TX12 по USB)

Пультом можно рулить дрона прямо из веб-панели — **без доп. софта на ноуте**. TX12
читается браузером через **Gamepad API**, а панель стримит положения стиков на Pi
(`/fcsticks`). Подходит и для режима MSP Override (тогда работают roll/pitch/yaw), и
для полного RX=MSP (плюс **газ** со стика).

1. На пульте включи режим **USB Joystick (HID)**: воткни TX12 в ноут кабелем, на
   экране «USB» выбери **Joystick** (EdgeTX). Проверка: открой `https://hardwaretester.com/gamepad`
   — стики должны двигать оси.
2. Открой веб-панель в **Chrome/Edge** (там есть Gamepad API; в некоторых браузерах
   нужно нажать кнопку на пульте, чтобы он «проснулся»).
3. Внизу панели (когда подключён FC) появится строка **axis/sign** и список текущих
   осей — подвигай стиками и посмотри, какая ось что делает. Выставь индексы осей и
   знак для **R**(roll)/**P**(pitch)/**Y**(yaw)/**T**(throttle). Дефолт `0 / 1 / 3 / 2`
   (EdgeTX AETR), знак pitch обычно `-1`.
4. Нажми **ARM**, затем **Sticks: ON** — панель начнёт слать положения стиков
   (25 Гц) для roll/pitch/yaw. Отпустил/выключил — стики в центр (вотчдог 0.4 с).
   Авто-слежение на это время уступает пульту.
5. **Газ.** По умолчанию газ остаётся на кнопках **THR ▼/▲** (мышью) — это стабильный
   удерживаемый сетпойнт. Хочешь газ **со стика** — включи **Stick thr: ON** (ось `T`).
   Стик-газ сглаживается (`FC_THR_SMOOTH`), чтобы дёрганая/глитчащая ось не роняла
   обороты; всё равно держи газ на стике плавно.

> Почему так: газ — абсолютная величина, и один битый кадр оси мгновенно ронял его в
> минимум (ты это и видел: 1137 → 1000 → обратно). Теперь стик-газ опционален и
> сглажен, а d-pad-газ (мышью) на него больше не «забивается».

> Сначала всё проверь **со снятыми пропеллерами**. Стики — ручной режим: он
> перебивает авто-слежение и удержание места.

### Удержание места против ветра (station-keeping) — кнопка HOLD

Кнопка **HOLD** на панели включает **визуальную стабилизацию положения**: сайдкар по
камере оценивает, куда «уплыла» картинка (оптический поток, phase-correlation), и даёт
встречный **roll/pitch**, возвращая дрон на место — то есть **сопротивляется сносу
ветром**. Это **экспериментально** и взаимоисключимо со слежением (включил HOLD —
слежение выключилось, и наоборот); ручные стики/d-pad всегда главнее.

- **Лучше всего работает с камерой, смотрящей ВНИЗ** — тогда снос земли виден как
  сдвиг кадра. С камерой вперёд гасит «уплывание картинки», но плохо видит ход
  вперёд/назад (это масштаб, а не сдвиг).
- Нужны **текстура и свет** (над однотонным полом/в темноте поток слепнет).
- Тюнинг: `FC_HOLD_KP` (резкость, µs/пиксель), `FC_HOLD_MAX` (лимит коррекции, µs),
  `FC_HOLD_RES` (рабочая ширина, px), `FC_HOLD_LEAK` (утечка накопленного сноса),
  `FC_HOLD_DEAD` (мёртвая зона, px), `FC_HOLD_INVERT_X/Y` (если толкает не в ту сторону).
- Эндпоинт: `/fchold?on=1|0`. Проверяй направления **со снятыми пропами**: качни дрон
  рукой — коррекция должна идти **навстречу** сносу; если по ходу — `FC_HOLD_INVERT_X/Y=1`.

> Это не GPS/позиционный hold — это «держать кадр на месте». Дрейф со временем
> возможен (поток шумит), поэтому `FC_HOLD_LEAK` слегка «забывает» накопленное.
> Полноценный позишн-холд — это уже GPS Rescue/iNav, отдельная история.

### Если в прошивке нет MSP Override

`MSP OVERRIDE` — это **опция компиляции Betaflight** (`USE_MSP_OVERRIDE`), и в стоковых
билдах для F4 её часто **нет**. Признак: во вкладке **Modes** нет строки
«MSP OVERRIDE». Тогда наши `MSP_SET_RAW_RC` просто игнорируются — дрон не реагирует
(это безопасно, но и не летит). Три пути:

**1) Вкомпилировать MSP Override (рекомендуется).** В Betaflight Configurator →
**Firmware Flasher** → выбери плату (`SPEEDYBEEF405V4` для SpeedyBee F405 V4) → раскрой
**build options / «Show options»** и поищи **MSP Override** (категория Other) → собери и
прошей. Если в облачном билдере этой опции нет — собери из исходников с
`USE_MSP_OVERRIDE` (F405 = 1 МБ флеша, фича крошечная и влезает; при нехватке места
отключи лишнее). После прошивки строка «MSP OVERRIDE» появится в Modes — дальше всё по
инструкции.

> SpeedyBee F405 V4 (STM32F405, гиро ICM42688P, есть баро) под это подходит; нужен только
> билд с `USE_MSP_OVERRIDE`.

**2) Полный RX = MSP (без пульта).** Pi становится приёмником целиком и рулит **всем**:
roll/pitch/yaw + **газ + арм**. Пульта-override нет. Это **реализовано отдельным
защищённым режимом** `FC_MODE=full` — подробная инструкция ниже (раскрывашка
«ПОЛНЫЙ RX=MSP»). Опаснее варианта 1 (нет ручного перехвата с пульта) — но газом ты
управляешь с панели, а при потере связи срабатывает авто-снижение и дизарм.

**3) Не городить override — управлять не дроном, а подвесом.** Если задача «вести цель
камерой», поставь SIYI-подвес: тот же lock-on крутит подвес, а дрон не трогаем. Куда
безопаснее.

<details>
<summary><b>⛔ ПОЛНЫЙ RX=MSP — подробная инструкция (что и как делать). Развернуть</b></summary>

> В этом режиме **нет пульта**. Ноутбук с веб-панелью становится твоим пультом: **газ**
> ты держишь кнопками THR на панели, **арм** — кнопкой ARM, **повороты/наклоны** —
> авто-слежение или d-pad. Делай это, только если осознаёшь риск. Все первые шаги — **со
> снятыми пропеллерами**.

#### Ф1. Прошей failsafe-логику в голове
Три уровня защиты, как они работают:
1. **Софт-ARM (панель).** Пока ARM = OFF, арм-канал шлёт «disarmed» — моторы не
   запустятся, что бы ни случилось.
2. **Потеря ноутбука.** Если панель замолчала дольше `FC_LINK_TIMEOUT` (по умолч. 3 с)
   — Pi сам **плавно сбрасывает газ** (`FC_DESCENT_RATE` µs/с) до минимума и **дизармит**.
3. **Смерть процесса на Pi.** Если сайдкар упал — поток MSP прекращается, и срабатывает
   **штатный failsafe Betaflight** (Stage 2). Поэтому failsafe в Betaflight обязателен.

#### Ф2. Настрой Betaflight (с компа)
1. **Ports** — на UART, к которому подключён Pi, включи **MSP**, 115200. (По USB MSP уже есть.)
2. **Включи MSP-RX** (это и есть «RX=MSP»: каналы берутся из MSP-сообщений). Надёжнее
   всего через **CLI**:
   ```
   feature RX_MSP
   feature -RX_SERIAL
   save
   ```
   (в Configurator — Receiver → выбрать **MSP**, если опция есть в твоей версии.)
3. **Modes**:
   - **ARM** повесь на канал **AUX1** (это `FC_CH_ARM=4`), диапазон «armed» — высокий
     (напр. 1800–2100). Pi будет слать туда 2000 (`FC_ARM_US`) / 1000 (`FC_DISARM_US`).
   - **ANGLE** (горизонт-режим) сделай **всегда включённым**: повесь на любой AUX и задай
     диапазон на весь ход, **или** включи его постоянно. Без angle дрон не самостабилизируется.
4. **Failsafe**: Stage 2 = **Drop** (проще всего) или Land. Проверь, что срабатывает,
   если выдернуть/заглушить MSP.
5. **Receiver-вкладка**: проверь, что каналы вообще двигаются от Pi (см. Ф4).
6. Проверь **порядок каналов** `map` (по умолчанию AETR → `FC_CH_ROLL/PITCH/THROTTLE/YAW`
   = 0/1/2/3). Если иной — выставь `FC_CH_*`.

#### Ф3. Запусти сайдкар в полном режиме
```bash
pip3 install pyserial ncnn numpy opencv-python
ND_PARAM=nanodet.param ND_BIN=nanodet.bin ND_INPUT=320 \
  YOLO_LABELS=coco.names YOLO_FILTER=person YOLO_SOURCE=rpicam \
  FC=betaflight FC_MODE=full FC_PORT=/dev/ttyAMA0 \
  FC_THROTTLE_MIN=1000 FC_THROTTLE_MAX=1700 FC_THR_STEP=25 \
  python3 tools/nanodet-sidecar/nanodet_ncnn_sidecar.py
```
В консоли — **красный баннер `FULL RX=MSP`**. На панели у кнопки ARM теперь подпись
**«ARM MOTORS»** и появляются кнопки **THR ▼ / THR ▲** с текущим значением газа.

#### Ф4. Проверка БЕЗ пропеллеров (обязательно, на столе)
> 🚨 **Пропеллеры сняты.** Подключи полётник к компу по USB и открой **Receiver-вкладку**.
1. Подай питание на полётник, открой `http://<ip-пи>:8080`.
2. Нажми **ARM MOTORS** (красная). В Receiver-вкладке канал **AUX1 (ARM)** должен прыгнуть
   в высокое значение; моторы (без пропов!) перейдут в idle.
3. Жми **THR ▲** несколько раз — канал **Throttle** в Receiver растёт ступеньками
   (`FC_THR_STEP`), моторы раскручиваются. **THR ▼** — вниз.
4. Нажми **пробел** и пройди перед камерой (или потыкай d-pad) — каналы **Yaw/Pitch**
   двигаются в нужную сторону (не туда → `FC_INVERT_YAW=1`/`FC_INVERT_PITCH=1`).
5. **Проверка failsafe-связи:** при поднятом газе **закрой вкладку браузера** (или
   выдерни Wi-Fi у ноута). Через ~3 с throttle должен поехать вниз и AUX1 → disarm.
6. Выключи **ARM MOTORS**, сними питание.

#### Ф5. Первый полёт
Открытая площадка, **никого вокруг**. Питание, панель открыта. Нажми **ARM MOTORS**,
плавно подними газ кнопками **THR ▲** до зависания. Включи **пробел** и кликни человека —
Pi ведёт цель по yaw/pitch, **высоту держишь ты газом**. Снижение — **THR ▼** до посадки,
потом **ARM MOTORS = OFF**. Аварийно: **ARM MOTORS OFF** (моментальный дизарм) или просто
закрой вкладку (сработает авто-снижение).

#### Ф6. Переменные этого режима
| Переменная | Что | По умолч. |
|---|---|---|
| `FC_MODE=full` | включить полный RX=MSP (иначе MSP Override) | `override` |
| `FC_CH_ARM` | индекс арм-канала (AUX1=4) | `4` |
| `FC_ARM_US` / `FC_DISARM_US` | значения «armed»/«disarmed» на арм-канале | `2000` / `1000` |
| `FC_THROTTLE_MIN` / `FC_THROTTLE_MAX` | пределы газа (MAX — потолок безопасности) | `1000` / `1700` |
| `FC_THR_STEP` | шаг газа на одно нажатие THR, µs | `25` |
| `FC_LINK_TIMEOUT` | сек тишины панели до авто-снижения | `3.0` |
| `FC_DESCENT_RATE` | скорость сброса газа при потере связи, µs/с | `150` |

#### Ф7. Полётник уходит в циклический ребут (звук перезагрузки ESC по кругу)

Почти всегда это **не** кадры MSP (они корректны), а одно из двух:

1. **Просадка питания (brownout).** Если FC питается **только от USB** (или слабого BEC),
   а к нему подключены ESC/моторы — при арме/добавлении газа бросок тока сажает 5 В, FC
   ребутится, ESC пищат, и так по кругу, пока не передёрнешь питание. **Лечение:** питай
   полётник **от аккумулятора** (или нормального 5 В BEC), а для стендовой проверки связи
   **отключи моторы** или держи газ на минимуме. Pi питай **отдельно**.
2. **Не вычитывались ответы FC (USB-VCP).** Каждый `MSP_SET_RAW_RC` полётник ACK-ает; если
   их не читать, на USB буфер забивается и FC может залипнуть/ребутнуться. **Исправлено** —
   сайдкар теперь читает и отбрасывает ответы. Обнови код (`git pull`).
3. **ModemManager/brltty хватает порт (главная причина по USB!).** Симптом-маркер: **всё
   работает, пока скрипт запущен, а как только закрываешь скрипт — FC уходит в бутлуп, и
   лечится только если выдернуть USB-кабель из Pi** (сброс питания FC не помогает). Это
   значит: как только наш скрипт отпускает `/dev/ttyACM*`, порт мгновенно хватает системный
   сервис **ModemManager** (реже **brltty**) — шлёт в FC AT-команды и дёргает DTR, FC
   ресетится, переподключается по USB, ModemManager хватает снова → петля.

   **Лечение (на Raspberry Pi, один раз):**
   ```bash
   # 1) запретить ModemManager трогать полётник (STM32 VCP = 0483:5740):
   sudo tee /etc/udev/rules.d/99-betaflight.rules >/dev/null <<'EOF'
   SUBSYSTEM=="tty", ATTRS{idVendor}=="0483", ATTRS{idProduct}=="5740", ENV{ID_MM_DEVICE_IGNORE}="1", ENV{ID_MM_PORT_IGNORE}="1"
   EOF
   sudo udevadm control --reload-rules

   # 2) проще и наверняка — если модем не нужен, вырубить сервис совсем:
   sudo systemctl mask --now ModemManager

   # 3) если стоит brltty (тоже ворует USB-serial) — снести:
   sudo apt-get purge -y brltty
   ```
   Переподключи USB. Сайдкар теперь и сам открывает порт **эксклюзивно** и не «гудит» DTR
   при закрытии, но ModemManager всё равно лучше убрать — это корень проблемы.

Как быстро отличить причины: **останови сайдкар (Ctrl-C)**. Если бутлуп **начинается
именно после закрытия** и лечится выдёргиванием USB — это **пункт 3 (ModemManager)**. Если
FC циклится **во время работы** и не лечится ничем, кроме снятия питания — это **brownout
(пункт 1)**, чини питание. На UART ещё проверь TX/RX и общий GND. Снизить нагрузку:
`FC_RATE=30`.

</details>

## Как это устроено

### Пайплайн инференса Android
```
Кадр CameraX ImageAnalysis
  ├── streamExecutor   → composeFrame() + нарисовать последние боксы → MjpegServer.pushFrame()   [~30 fps]
  └── inferenceExecutor → YOLO / ONNX инференс → обновить lastKnownDets                          [скорость инференса]
```
MJPEG-сервер работает на полной скорости захвата; инференс выполняется независимо и не замедляет трансляцию.

### MJPEG-сервер
`MjpegServer` использует обычный `ServerSocket`. Каждый клиент обслуживается в отдельном потоке из кэшированного пула. `pushFrame()` кодирует JPEG один раз в `AtomicReference<ByteArray>` — каждый клиентский поток читает последний кадр независимо, без лишнего кодирования.

### Letterbox-предобработка
Кадры дополняются серым цветом (114) до квадрата, соответствующего размеру входа модели. Координаты детекций обратно преобразуются в пиксельное пространство оригинального кадра перед отображением.
